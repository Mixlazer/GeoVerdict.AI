# Product Proposal — GeoVerdict.AI

## 1. Обоснование идеи

GeoVerdict.AI решает прикладную задачу выбора офлайн-локации для нового магазина или точки продаж. Для малого бизнеса это решение обычно принимается вручную: предприниматель смотрит карту, сравнивает конкурентов, оценивает поток “на глаз” и рискует ошибиться без объективной картины.

Проблема особенно остра в сценариях быстрого запуска:

- франчайзи должен выбрать одну из нескольких точек в незнакомом районе;
- собственник кафе или аптеки хочет понять, стоит ли переплачивать за “видимое” место;
- менеджер развития сети проверяет гипотезу на десятках адресов без доступа к дорогим GeoBI-платформам.

Ценность GeoVerdict.AI в том, что система не просто показывает слои карты, а выдает объяснимый вердикт и предлагает альтернативу рядом, если исходная точка слабая.

## 2. Цель и метрики

### Цель проекта

Собрать PoC мультиагентной системы, которая по координатам и типу бизнеса формирует понятный вердикт о привлекательности локации для открытия магазина.

### Продуктовые метрики

- `Time-to-verdict < 45s` для одной точки при нормальной доступности внешних сервисов.
- `>= 80%` тестовых пользователей считают объяснение достаточным для первичного решения.
- `>= 60%` анализов завершаются рекомендацией действия: выбрать точку, сдвинуться или отказаться.

### Агентские метрики

- доля анализов без ручного вмешательства `>= 95%`;
- доля успешного завершения пайплайна `>= 97%`;
- доля случаев, когда optimizer предлагает улучшенную точку при реальном выигрыше `>= 70%` на эталонных сценариях.

### Технические метрики

- `p95 latency` полного пайплайна `< 60s`;
- `p95 latency` API чтения результата `< 500ms`;
- error rate по внешним гео-запросам `< 5%` с фолбеком;
- стоимость анализа в облачном LLM-режиме `<= $0.05`.

## 3. Сценарии использования

### Базовые сценарии

1. Предприниматель выбирает тип бизнеса и ставит точку на карте.
2. Система собирает контекст района, конкурентов и косвенные признаки трафика.
3. Пользователь получает итоговый verdict и список ключевых рисков.

### Сценарии развития

- сравнение нескольких соседних точек в одном районе;
- повторная проверка после смены бизнес-типа;
- передача результата менеджеру развития как отчета для быстрой сверки.

### Edge cases

- плохое покрытие OSM или Nominatim по конкретной точке;
- слишком мало конкурентов в радиусе, что снижает confidence;
- точка находится внутри квартала без явной уличной доступности;
- нестабильный внешний LLM или отсутствие ключа;
- пользователь выбирает точку в городе вне первичного фокуса PoC.

## 4. Ограничения

### Технические ограничения

- p95 полной обработки должен укладываться в 60 секунд;
- внешний гео-стек ограничен rate limits публичных OSM/Nominatim API;
- context budget LLM ограничен агрегированными признаками, а не “сырыми” POI-списками;
- observability должна покрывать request id, шаги агентов, latency, cost, fallback.

### Операционные ограничения

- PoC должен запускаться локально через Docker Compose;
- бюджет на облачные вызовы должен оставаться низким;
- система должна работать даже без ключей LLM за счет эвристического fallback-режима;
- качество объяснений важнее, чем глубина геомаркетинговой аналитики уровня enterprise.

## 5. Архитектурный набросок

Система состоит из трех больших частей:

- пользовательское веб-приложение с картой и отображением verdict;
- backend-orchestrator с набором агентов и гео-утилит;
- отдельный ops dashboard для наблюдаемости, качества и стоимости.

Ключевые интеграции:

- OSM / Overpass для POI и транспорта;
- Nominatim для reverse geocoding;
- LLM Router для summary/fallback-провайдеров;
- Prometheus / Grafana для runtime-метрик.

## 6. Потенциальный data flow

1. Frontend отправляет координаты, город и тип бизнеса.
2. Backend создает analysis request и переводит его в статус `processing`.
3. Geo-Agent получает адрес и признаки улицы.
4. Competitor-Agent выделяет релевантных конкурентов.
5. Traffic-Agent оценивает косвенный поток.
6. Analyst-Agent собирает итоговый `LocationScore`.
7. Optimizer оценивает точки вокруг и, если нужно, формирует `OptimizationSuggestion`.
8. Verdict composer выдает итог, риски и объяснение.
9. Результат сохраняется, а ops dashboard получает агрегаты по latency/cost/quality.

### Что делегируется агентам / LLM

- генерация читаемого объяснения и вердикта;
- нормализация неполных данных в понятные бизнес-признаки;
- структурированная сводка рисков и рекомендаций.

### Что не делегируется LLM

- расчет расстояний и геометрии;
- запросы к гео-API;
- базовый скоринг и guardrails;
- расчеты SLA, лимитов и observability-метрик.
