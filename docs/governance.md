## Политика логов

- Каждый HTTP-запрос получает `request_id`.
- В логи попадают технические события: latency, шаги агентов, тип ошибок, провайдер.
- Не логируются пользовательские секреты и внешние токены.
- Координаты и адреса логируются только как часть анализа и только для воспроизводимости PoC.
- Trace-данные должны быть доступны команде разработки и MLOps, но не анонимным пользователям.

## Политика персональных данных

PoC не требует обязательной регистрации и не обрабатывает чувствительные персональные данные как часть основного сценария. Если в дальнейшем будет добавлена аутентификация:

- email и role должны храниться отдельно от payload анализа;
- аналитические данные должны быть минимизированы;
- retention-период для user-linked traces должен быть ограничен;
- экспорт или удаление пользовательских данных должен быть возможен на уровне сервиса.

## Защита от injection и небезопасных действий

- LLM получает нормализованные признаки, а не произвольный HTML или длинные сырые ответы API.
- Вся структурированная генерация проходит Pydantic-валидацию.
- Вердикт не может менять исходные численные признаки: только интерпретировать их.
- Любые административные ops endpoint защищаются заголовком `X-Ops-Token`.
- Действия с побочными эффектами в будущем должны требовать подтверждения пользователя.

## Guardrails

- если геоданные неполные, confidence автоматически снижается;
- если конкуренты не определились надежно, система пишет об этом явно;
- если optimizer не находит прироста > 15%, он не предлагает смещение;
- если внешние API упали, PoC работает в режиме эвристического fallback, а не “придумывает” уверенный ответ.
